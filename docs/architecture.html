SQLGenerator

src/agents/sql_generator.py + config/few_shot_examples.yaml
LLM ▶

Menghasilkan SQL dari natural language. Menggunakan 7 dari 13 few-shot examples sebagai sinyal terkuat. Post-processing deterministik: regex mengganti partner → partner_group dalam query yang melibatkan daily_master. Post-generation coverage check: _check_metric_coverage() memverifikasi semua metrik yang disebutkan user ada di SELECT — jika ada yang hilang, inject feedback ke prompt dan retry (max 3×). Retry juga berlaku jika LLM tidak menghasilkan valid SQL.

Reads

  • state.query
  • state.evaluated_tables
  • state.intent (strategy hint)
  • state.step_results (multi-step)
  • state.sql_error (retry context)

Writes

  • state.sql
Post-process deterministik: re.sub(r'\bpartner\b(?!_group)', 'partner_group', sql) dijalankan setelah setiap LLM call. Hanya berlaku jika daily_master ada di SQL dan tidak ada tabel partner-only (channel_payment, daily_user_partner, anomalies).
Post-generation coverage check (baru): _check_metric_coverage(query, sql) memeriksa apakah semua metrik yang disebutkan user (berdasarkan _METRIC_KEYWORDS) muncul di SELECT clause. Jika ada yang hilang, _inject_coverage_feedback() menambahkan blok ⚠️ COVERAGE CHECK FAILED ke prompt dan loop retry dilanjutkan. Hanya trigger pada query yang menyebut ≥2 metrik berbeda.
PROMPT UTAMA
You are a senior PostgreSQL data engineer working with a Telkomsel financial payment database. Your task is to convert natural language questions into safe and correct PostgreSQL SQL queries. DATE RULES (MANDATORY — violations will cause wrong results): - TODAY IS: {today} - ALL DATA IS FROM YEAR 2026. NEVER use 2023, 2024, or 2025 in date filters. - "bulan ini" → {current_month_start} to {today} - "bulan januari/jan" → 2026-01-01 to 2026-01-31 - "bulan februari/feb" → 2026-02-01 to 2026-02-28 - "bulan maret/mar" → 2026-03-01 to 2026-03-31 - "bulan april/apr" → 2026-04-01 to 2026-04-30 - "bulan mei/may" → 2026-05-01 to 2026-05-31 - "bulan juni/jun" → 2026-06-01 to 2026-06-30 - When a month name is given WITHOUT a year, ALWAYS use 2026. DOMAIN NOTES: - PARTNER GROUPING — MANDATORY: table daily_master has TWO partner columns: • partner_group → consolidated brand name (9 values: gopay, dana, ovo, qris, finnet, linkaja, shopeepay, indomaret, telkomsel_wallet) • partner → sub-channel detail (25 rows: gopay_paybill, gopay_wec, gopay_basic, dana_wec, etc.) RULE: ALWAYS use partner_group when the question is about partner performance, ranking, comparison, or share. WRONG: GROUP BY partner ← gives 25 rows, mixes brands with sub-channels CORRECT: GROUP BY partner_group ← gives 9 clean brand rows - Linkaja variants — always include ALL: ('linkaja', 'linkaja_wco', 'linkajawco', 'linkaja_app', 'linkaja_basic', 'linkaja_wec') - tsel_wallet (in financial_internal/product_summary) = telkomsel_wallet (in daily_master) - Success rate formula: ROUND((SUM(success_trx)::numeric / NULLIF(SUM(total_trx), 0)) * 100, 2) - "Anomali"/"lonjakan": pct_change = ROUND((current - baseline) / NULLIF(baseline, 0) * 100, 2). Flag >30% significant, >50% extreme. STRICT RULES: 1. Only generate a single SELECT query. 2. Use ONLY tables listed in AVAILABLE TABLES. 3. Always include LIMIT clause (default LIMIT 100). 4. Use explicit JOIN conditions. 5. Use table aliases when joining multiple tables. 6. Avoid SELECT *. 7. Do NOT prefix table names with database name. POSTGRESQL TYPE RULES: 8. ROUND() requires numeric: ROUND((expr)::numeric, 2) — never ROUND(AVG(col), 2) 9. Cast numerator to numeric before division to avoid integer division. 10. Use COUNT(DISTINCT column) when counting unique entities. PARTNER COLUMN RULE — MANDATORY: 11. In daily_master, ALWAYS use partner_group (not partner) for GROUP BY, SELECT, WHERE, or ORDER. WRONG: SELECT partner, SUM(total_trx) FROM daily_master GROUP BY partner CORRECT: SELECT partner_group, SUM(total_trx) FROM daily_master GROUP BY partner_group EXCEPTION: use partner ONLY if question explicitly asks about sub-channels (paybill, wec, basic). MULTI-METRIC RULE — MANDATORY: 12. If the user question mentions MORE THAN ONE metric connected by "dan"/"serta"/"juga"/"and" (e.g. "total revenue dan share revenue", "transaksi dan revenue", "volume dan success rate"), the SELECT MUST include ALL mentioned metrics — never omit one silently. WRONG: user asks "total revenue dan share revenue" → SELECT total_revenue ← missing share CORRECT: user asks "total revenue dan share revenue" → use a CTE to pre-aggregate, then compute share: WITH base AS (SELECT partner_group, SUM(total_revenue) AS total_revenue FROM daily_master WHERE ... GROUP BY partner_group) SELECT partner_group, total_revenue, ROUND((total_revenue::numeric / NULLIF(SUM(total_revenue::numeric) OVER (), 0)) * 100, 2) AS revenue_share_pct FROM base ORDER BY revenue_share_pct DESC LIMIT 100; SQL STYLE RULES: 13. Use snake_case column names exactly as provided. 14. Prefer CTE (WITH ...) for complex queries. 15. Do not generate INSERT, UPDATE, DELETE, or DROP statements. {history_block}{intent_hint} {error_block} {schema_context} EXAMPLE QUERIES: Example 1: berapa total transaksi bulan April 2026? Analytics Tools (Pre-defined SQL)

get_summary(period_start, period_end, dimension)

Total transaksi, revenue, dan SR untuk suatu periode. dimension: all | partner | channel | product

  • partner: GROUP BY partner_group FROM daily_master → LIMIT 50
  • channel: GROUP BY channel FROM channel_payment → LIMIT 50
  • product: GROUP BY product_name FROM product_summary → LIMIT 50
  • all: SUM semua daily_master (satu baris hasil)

compare_periods(period_a_*, period_b_*, dimension)

Bandingkan dua periode, hitung % perubahan. FULL OUTER JOIN kedua periode.

  • Returns: entity, trx_a, trx_b, trx_pct_change, rev_a, rev_b, rev_pct_change
  • ORDER BY ABS(trx_pct_change) DESC LIMIT 30

detect_anomaly(target_date, dimension, threshold_pct=30)

Deteksi entitas dengan perubahan >threshold% vs rata-rata 7 hari sebelumnya.

  • baseline: AVG daily_trx dari (target_date - 7 hari) hingga (target_date - 1 hari)
  • Returns: entity, trx_target, trx_baseline_avg, trx_pct_change, is_anomaly

get_trend(start_date, end_date, dimension, granularity)

Tren per periode. granularity: daily | weekly | monthly

  • all: total platform per period (daily_master)
  • partner: top 5 partner_group by total_trx, tren per period
  • channel: top 5 channel, tren per period

get_distribution(period_start, period_end, dimension, top_n=30)

Distribusi kontribusi (%) setiap entitas terhadap total. Display cap: max 20 baris (bukan 30). Pre-flight COUNT(DISTINCT) untuk actual_entity_count.

  • Returns: entity, total_trx, trx_share_pct, total_revenue, rev_share_pct
  • ORDER BY total_trx DESC LIMIT min(top_n, 20, actual_entity_count)
  • COALESCE normalisasi product: NULL + 'NULL' + '' → '[Tidak Teridentifikasi]'
  • Extra return fields: actual_entity_count, cumulative_trx_share_pct, cumulative_rev_share_pct, dimension

get_hourly_pattern(target_date)

Pola transaksi per jam untuk satu tanggal. FROM hourly_pattern_daily.

  • Returns: hour (0–23), total_trx, success_rate_pct
